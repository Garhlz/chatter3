# Chatter3 前端 UI 设计说明

本文档记录当前 Tauri / React 客户端的界面原则、组件边界和交互约定。它既是后续迭代的依据，也可以作为阅读前端代码时的路线图。

## 1. 设计目标

当前界面的定位是：**原生桌面感、消息优先、适度展示技术状态的现代聊天客户端**。

- 原生桌面感：使用系统字体、稳定的双栏布局、克制的圆角和阴影，不模拟网页仪表盘。
- 消息优先：主视线应落在会话和消息上；连接状态、开发工具和群管理按需出现。
- 技术状态适度可见：断线、重连、发送失败必须让用户知道，但正常连接不反复弹出提示。
- 可学习：组件和状态职责保持直接，避免为了少写几行代码而引入难追踪的抽象。

## 2. 信息架构

登录后的窗口由两个长期区域和若干临时浮层组成：

```text
┌──────────────────┬────────────────────────────────────────┐
│ 会话侧栏 304 px   │ 当前聊天                               │
│                  │ ┌────────────────────────────────────┐ │
│ 品牌 / 新建群聊   │ │ 会话身份、状态与少量操作             │ │
│ 搜索              │ ├────────────────────────────────────┤ │
│ 会话列表          │ │                                    │ │
│                  │ │ 居中、最大 920 px 的消息时间线       │ │
│                  │ │                                    │ │
│ 当前用户 / 设置   │ ├────────────────────────────────────┤ │
│                  │ │ 附件、自动增高输入框、发送           │ │
└──────────────────┴────────────────────────────────────────┘
```

- `Sidebar` 只负责会话导航和当前账号入口。
- `ChatPanel` 负责当前会话的标题栏、消息区域、输入区和群信息抽屉。
- Toast、登录失效横幅、模态框和图片预览位于应用根节点，不挤压主布局。
- 群成员等低频信息从右侧抽屉按需出现，避免长期占用消息宽度。

当窗口宽度低于 `820px` 时，侧栏改为覆盖层。这里的目标是兼容用户缩窄桌面窗口，并不是把第一阶段做成完整的手机端产品。

## 3. 视觉系统

### 3.1 主题

继续保留两套现有主题，并统一通过 `styles/tokens.css` 中的 CSS 变量表达：

- `Latte`：暖灰背景、紫蓝强调色、白色消息气泡，适合日常使用。
- `One Dark`：延续开发者熟悉的深灰底色，并提高文字和边框对比度。

组件不应直接写主题专用颜色。新增颜色时，先判断它是否是一个可复用的语义角色，例如 `--color-danger-soft`，再由主题分别提供具体值。

### 3.2 密度与层级

- 会话项的基础高度为 `64px`，同时容纳标题、时间、摘要和未读数。
- 聊天标题栏高度为 `68px`，只保留身份、连接异常和不超过三个主要操作。
- 消息正文最大宽度为 `920px`，宽屏时不会被拉成难读的长行。
- 常驻界面主要依靠背景色和 1px 边框分层；阴影仅用于菜单、抽屉、模态框和 Toast。
- 图标统一使用 `lucide-react`，普通操作按钮不再使用难以辨认的文本符号。

## 4. 组件职责

| 组件 | 职责 | 不应承担的职责 |
| --- | --- | --- |
| `App` | 应用壳、窗口级浮层、恢复会话和桌面事件监听 | 消息协议解析 |
| `Sidebar` | 组合品牌、会话列表和用户栏 | 保存会话数据 |
| `ConversationList` | 搜索、选择和展示会话摘要 | 请求聊天历史 |
| `ChatPanel` | 当前会话标题栏和详情抽屉开关 | 直接操作 WebSocket |
| `MessageList` | 分组时间线、滚动、附件展示和重试入口 | 生成协议 payload |
| `Composer` | 当前会话草稿、IME 输入、附件选择和发送触发 | 维护连接生命周期 |
| `GlobalFeedback` | 全局成功/错误反馈和登录失效提示 | 业务错误恢复策略 |
| `chatStore` | 前端状态和业务动作的协调 | Tauri 窗口渲染细节 |

`ui/Avatar.tsx`、`ui/IconButton.tsx` 和 `ui/ConnectionIndicator.tsx` 是小型基础组件。它们只封装稳定、重复的外观与无障碍规则，不建立额外的“大而全”组件库。

## 5. 消息时间线规则

`buildMessageTimeline` 是纯函数，负责把扁平消息转成日期分隔线和消息组：

1. 跨本地自然日时插入日期分隔线。
2. 相邻消息发送者相同、时间顺序正确，且间隔不超过 5 分钟时合为一组。
3. 自己的消息靠右，其他人的消息靠左并显示头像和发送者名称。
4. 正常的“已发送”状态不显示图标；发送中和发送失败才出现状态。
5. 文件消息使用独立文件气泡，图片可鉴权加载后预览；下载仍通过 HTTP / Tauri 文件能力完成。

这种转换和 JSX 分离后，可以不启动浏览器就测试分组边界，也能避免渲染组件中出现多层条件判断。

## 6. 滚动与新消息

- 每个会话单独保存草稿和滚动位置，切换会话后可以继续原来的上下文。
- 用户位于底部附近时，新消息自动跟随到底部。
- 用户正在阅读旧消息时，不抢走滚动位置，而是显示“新消息”悬浮按钮。
- 加载更早历史后，用加载前后的 `scrollHeight` 差值补偿滚动位置，防止内容突然跳动。
- 历史加载目前保留显式按钮，比自动触底加载更容易理解和排查。

## 7. 输入和附件

- `Enter` 发送，`Shift + Enter` 换行。
- 输入法正在组合中文时不响应 Enter 发送，避免拼音候选确认误发消息。
- 输入框随内容增高，最多约六行，超过后内部滚动。
- 草稿以会话 ID 为键存储。异步发送完成时只清空当时发送的会话，避免快速切换会话误删另一个草稿。
- 当前协议没有群文件上传语义，因此群聊附件按钮明确禁用。前端不应偷偷退化成公共文件上传，否则 UI 看似成功但消息会进入错误会话。

## 8. HTTP、WebSocket 与 Tauri 边界

- HTTP：登录、历史、群管理、资料和文件上传/下载等请求-响应操作。
- WebSocket：实时消息、在线状态、确认和心跳等长连接事件。
- Tauri Rust 壳：桌面窗口、系统文件保存、本地数据库和桌面事件桥接。
- React / Zustand：界面呈现、草稿、滚动位置、连接状态显示和用户操作编排。

聊天协议本身仍在 TypeScript realtime client 和 Go 服务端之间对齐。UI 重构没有新增 Go API、WebSocket 事件或 Tauri command。

## 9. 样式文件导航

| 文件 | 内容 |
| --- | --- |
| `styles/tokens.css` | 颜色、尺寸、圆角、阴影和主题变量 |
| `styles/base.css` | 重置、表单、按钮、头像和基础工具类 |
| `styles/shell.css` | 登录页、应用双栏壳、侧栏和聊天标题栏 |
| `styles/chat.css` | 时间线、气泡、附件、输入区和群详情抽屉 |
| `styles/overlays.css` | 模态框、资料页、开发面板、Toast 和图片预览 |
| `styles/responsive.css` | 紧凑窗口、侧栏覆盖层和减少动态效果 |

入口 `styles.css` 只按依赖顺序导入这些文件。响应式规则必须最后加载，才能以较低的维护成本覆盖基础布局。

## 10. 验证方式与后续迭代

前端基础验证：

```bash
cd frontend
npm test
npm run typecheck
npm run build
```

Tauri 壳层有变化时还应执行：

```bash
cd frontend/src-tauri
cargo check
```

下一阶段可继续补充：

- 会话置顶、静音和右键菜单；
- 更完整的键盘导航和焦点回到触发按钮；
- Toast 队列而不是单条全局字符串；
- 群文件协议设计完成后的附件能力；
- Storybook 或独立组件预览页，用于主题和状态矩阵的视觉回归；
- 更接近真实 Store 行为的 Composer、断线重连和滚动交互测试。
